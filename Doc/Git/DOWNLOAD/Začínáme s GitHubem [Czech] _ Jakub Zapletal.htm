Začínáme s GitHubem [Czech]

April 9, 2013 / Jakub Zapletal / 1 Comment

Jak název článku napovídá, budeme se věnovat práci s GitHubem (https://github.com/), ale velká část bude pojednávat o práci s Gitem jako takovým. Pokud vám zde budou chybět některé informace okolo Gitu, tak doporučuji český překlad knihy, která je zdarma (http://git-scm.com/book/cs), případně ohledně GitHubu jejich oficiální help (https://help.github.com/).

Co je Git

Git je verzovací systém původně navržený pro správu verzi operačního systému Linux a nyní je nejrozšířenějším verzovacím systémem nejen u vývojářů webových aplikací. Díky tomu, že byl navržen pro OS Linux, tak také pod tímto OS má nejlepší podporu. Uživatelé Windows a XAMPPu se nemusejí bát, i pro ně existuje způsob, jak s ním mohou pracovat, aniž by měli nainstalovaný Linux.

Proč verzovat

Využívat verzovací systém, případně Git, není jen moderním hitem a být “IN”, ale opravdu vám to zkvalitní práci a můžete tak předejít spoustě problémům se ztrátou dat nebo jejich přepsáním. Využívání verzovacího systému je při práci v týmu vývojářů snad nutností, ale i jako samostatný vývojář naleznete výhody.

Práce v týmu

Zkoušeli jste někdy pracovat v týmu a nahrávat soubory na jedno místo například pomocí FTP? Já před X lety ano, a věřte, že je to peklo. Každou chvíli řešíte problém s přepsáním nechtěných souborů, většinou nelze dohledat, kdo problém zavinil,  a na možnost získání starší verze nějakého souboru, nebo dokonce celé aplikace, můžete zapomenout. Umožňuje vám to krásně rozdělovat funkční verzi aplikace od upravované verze, nedodělané verze, nebo si jen tak můžete vyzkoušet nějakou změnu a pak to zahodit.

Práce samostatného vývojáře

Pokud se zamyslíte nad výhodami práce v týmu, tak je můžete uplatnit i když pracujete sám, s tím rozdílem, že víte, kdo problém zavinil :-). Mimo to se často setkávám se situací, kdy pracuji na projektu z více míst (kancelář, doma) a potřebuji si hotovou práci přenášet z jednoho místa na druhé. V tomto vám verzovací systém také lehce pomůže, stačí jen odeslat vaši rozdělanou práci a v dalším místě si aktualizujete soubory – vše otázkou pár vteřin. Neocenitelnou výhodou je hlavně zálohování dat. Když přijdete o vaše data (poškodí se vám HDD), tak vše máte stále uložena na GitHubu.

Co je GitHub

GitHub je online služba, která vám umožní si vaše data v Gitu ukládat na online server. Máte je tak k dispozici všude, kde je přístupný internet. Mimo to máte kompletní správu vašich repozitářů (viz níže) a vašeho účtu přes webové rozhraní. Nabízí jak placený přístup, tak zdarma. Pokud chcete mít repozitáře skryté před veřejností, tak si musíte za balík potřebných repozitářů připlatit. Pokud vám nevadí, že si vaše kódy někdy může vyhledat a případně stáhnout, tak to máte zdarma. Existují projekty podobné GitHubu, ale zatím nedosahují jeho kvalit, jako příklad mohu uvést Bitbucket (https://bitbucket.org/).

Slovník pojmů

Nelze zde vypsat přehled všech pojmů, to by bylo na samostatný článek, a tak zde uvedu jen ty podstatné pro tento článek a ostatní si můžete dohledat zde: http://www.fuzzy.cz/cs/clanky/git-kapitola-11-git-slovnicek/

repozitář (repository) – Zaštiťuje celý 1 projekt, včetně všech souborů, verzí souborů, větví, úkolů a přístupů.

větev (branch) – Je to vývojová linie vašeho projektu. Paralelně jich povětšinou existuje více. Například si to představte tak, že 1 větev je verze vašeho systému, která je plně funkční a nasazena v ostrém provozu, a ve druhé větvi implementujete nové funkce, nebo opravy. Jakmile je práce na dané větvi dokončena, obvykle ji pak sloučíte s vaší hlavní (master) větví.

hlavní (master) větev – Takto označovaná větev obvykle obsahuje vaši produkční verzi projektu, kterou aktuálně používáte v ostrém provozu. Ve většině případů je právě tato větev výchozí větví pro vytváření nových větví.

commit – Uložení provedených úprav a vytvoření skupiny změn na daných souborech. Každé toto uložení vyžaduje váš komentář, například označení co se změnilo.

push – Odeslání všech posledních provedených commitů na server (GitHub).

pull – Aktualizace vašich lokálních souborů ze serveru (GitHub). Vždy to aktualizuje pouze tu větev, pro kterou chcete aktualizaci provést.

merge – Sloučení obsahu větví.

Instalace Gitu

Pro Linux i Windows viz http://git-scm.com/book/cs/%C3%9Avod-Instalace-syst%C3%A9mu-Git.

Dále je potřeba provést nastavení vaší identity apod., viz http://git-scm.com/book/cs/%C3%9Avod-Prvn%C3%AD-nastaven%C3%AD-syst%C3%A9mu-Git.

GitHub nyní nově nabízí vlastní desktopovou aplikaci pro uživatele Windows. Vzhled aplikace se drží grafické konvence Windows 8. Kromě grafického rozhraní nabízí také klasický přístup přes command line. Pokud se rozhodnete využívat tuto aplikaci, následující kapitoly článku berte jako informativní pro upřesnění problematiky. Většinu níže uvedené práce provedete v aplikaci pouhým klikáním.
Ke stažení zde: http://windows.github.com/

Vytvoření repozitáře

Před vytvořením repozitáře je nutné se registrovat na GitHubu, pokud jste to ještě neprovedli, tak učiňte tak nyní (https://github.com/).

Nový repozitář

Pokud jste nyní přihlášeni, tak nahoře vedle svého loginu je hned první ikona pro možnost vytvoření nového repozitáře.

Jako “Owner” vyberte svůj účet (měl by být automaticky vybrán).

Do “Repository name” napište název vašeho repozitáře. Jak vám GitHub radí, je vhodné používat krátké a výstižné názvy repozitářů.

Máte také možnosti si vyplnit kratší popis repozitáře do políčka “Description”.

Další důležitým nastavením je, zda daný repozitář chcete “Public” nebo “Private” (pouze pro placené účty).

Pokud zatrhnete “Initialize this repository with a README”, vytvoří se vám v novém repozitáři automaticky soubor README.md. Díky tomu, že repozitář bude obsahovat již nějaký soubor, bude možné s repozitářem plně pracovat, v opačném případě si tam musíme první soubory nahrát ručně sami (viz postup popsán níže).

Volba “.gitignore” vám zajistí, že určité soubory/složky nebudou Gitem verzovány, tedy nebudou se Gitem odesílat na server, ani se ze serveru nebudou aktualizovat. Pokud zatím nemáte představu, zda něco vybrat, případně co vybrat, tak danou možnost přeskočte.

Dáme “Create repository” a máme vytvořený repozitář.

SSH klíč

GitHub a i jiné servery pro sdílení Git repozitářů vyžadují kvůli zabezpečení, abyste s nimi komunikovali za pomocí ověřeného SSH klíče. Tento klíč funguje jako vaše identifikace k vašemu účtu. Když pak budete chtít pracovat se soubory ve vašem repozitáři, tak díky tomuto klíče si GitHub zjišťuje, že jste to doopravdy vy a nevydává se za vás někdo jiný. Návod naleznete přímo na stránkách GitHubu (https://help.github.com/articles/generating-ssh-keys).

Pozn.: Pokud používáte Windows aplikaci od GitHubu, SSH klíč nastavovat nepotřebujete, aplikace si jej vytvoří a nastaví automaticky sama.

Práce s repozitářem

Odeslání prvních souborů do repozitáře

Spustíme/otevřeme si command line (příkazovou řádku) a pomocí procházení adresářovou strukturou se dostaneme do adresáře, v kterém chceme mít požadovaný projekt. V adresáři si vytvoříme textový soubor s název README.md a do něj si napíšeme například “Můj první repozitář”.

Vytvoříme z adresáře pracovní složku pro Git:
git init

Řekneme Gitu, aby zaznamenal nové/změněné soubory:
git add -A

Vytvoříme skupinu změn:
git commit -m "first commit"

Nastavíme repozitář na GitHubu jako náš výchozí:
git remote add origin git@github.com:login_uzivatele/nazev_repozitare.git

Odešleme všechny commity na GitHub:
git push origin master
– “origin” jsme si výše definovali jako náš výchozí server a repozitář, “master” je název větve, který si git automaticky vytváří jako defaultní

Nyní, když se podíváte na GitHubu na svůj repozitář, tak vidíte odeslané soubory. Pokud je součástí vašich souborů README.md, tak jeho obsah vidíte pod výpisem souborů.

Stažení již existujícího repozitáře

Pomocí procházení adresářovou strukturou se dostaneme do adresáře, v kterém chceme mít umístěnou složku s požadovaným projektem. Složku můžete, ale nemusíte předem vytvářet.

Stažení existujícího repozitáře:
git clone git@github.com:login_uzivatele/nazev_repozitare.git nazev_slozky
– cestu upravte dle cesty k repozitáři na GitHubu, “nazev_slozky” přejmenujte dle toho, jak chcete mít/máte pojmenovaný adresář, kde bude umístěn projekt

Vytváření větví

Seznam větví zjistíte zadáním: git branch , kde právě aktuální větev je označena symbolem hvězdičky (*).

Pokud chcete vytvořit novou větev, tak počítejte s tím, že nová větev bude obsahovat přesně takový stav projektu/souborů jako větev, ze které ji vytváříte. Jste-li tedy aktuálně na “master”, tak nová větev bude obsahovat to samé co “master”. Pro vytvoření větve zadejte:
git checkout -b nazev_vetve
– to nám vytvoří novou větev a zároveň nám ji označí jako aktuální, na které zrovna pracujeme.

Pokud nyní znovu zadáte git branch , tak uvidíte 2 větve, “master” a nově vytvořenou, kde u nově vytvořené bude symbol *.

Jakmile nyní provedeme příkaz git push origin nazev_vetve , tak se nám na GitHubu vytvoří tato větev a objeví se tam všechny provedené úpravy. Na GitHubu se můžete přesouvat mezi všemi vytvořenými větvemi, které jste tam odeslali a sledovat změny v souborech.

Přepnutí větve

Pokud se chceme přepnout na jinou větev, stačí zadat git checkout nazev_vetve . Pozor, pokud jste v původní větvi upravili nějaké soubory, u kterých jste neprovedli operaci commit, tak změny v souborech se neprojeví v původní větvi, ale přenesou se zároveň s vámi do přepnuté větve.

Sloučení větví

Pokud máme v nějaké větvi změny a chceme, aby se změny zanesli do aktuální větve, zvolíme příkaz:
git merge nazev_jine_vetve

Nemusíme se bát nežádoucímu přepsání souborů. Git merguje inteligentně a pokud si náhodou neví s něčím rady, tak vám o tom dá vědět a vyžádá, abyste se sloučení provedli ručně, viz hláška:

Shell
1
2
3
Auto-merging README.md
CONFLICT (content): Merge conflict in README.md
Automatic merge failed; fix conflicts and then commit the result.

Git vám přímo řekne, v kterých souborech se objevil konflikt a co máte udělat, tedy vyřešit konflikt v souboru a pak commitnout změnu. Pokud při mergování mělo dojít ke sloučení více souborů, tak ostatní soubory, u kterých nenastal konflikt, byly bez problému mergnuty a nyní jen stačí vyřešit soubory, v kterých konflikt nastal. Seznam souborů, které nebyly v pořádku mergnuty, můžete zobrazit příkazem:
git status

Pokud si nyní otevřeme soubor README.md, tak Git nám tam označil konflikt takto:

README.md
1
2
3
4
5
6
Můj první repozitář
<<<<<<< HEAD
CCC
=======
AAA
>>>>>>> master

To co je obaleno mezi  <<<<<<< HEAD a  ======= označuje změny, které máte ve vaší aktuální větvi, a to co je obaleno mezi ======= a >>>>>>> master označuje změny, které jsem se pokusil mergnout z větve “master”.

Předpokládejme, že požadujeme úpravu v naší aktuální větvi za tu správnou. Upravíme soubor tak, aby vypadal správně takto:

README.md
1
2
Můj první repozitář
CCC

Nyní už jen stačí spustit příkazy:

Shell
1
2
git add -A
git commit -m "text informujici o oprave"

Pozn.: Pokud máte v konfliktu více souborů, doporučuji místo git add -A použít po každém ručně upraveném souboru příkaz git add nazev_souboru . Tímto způsobem budete moct průběžně příkazem git status sledovat, které soubory ještě musíte upravit. V případě nechtěného použití git add -A se všechny soubory označí, aniž byste je upravili a již pomocí git status nezjistíte, které soubory je potřeba upravit kvůli konfliktu.

Aktualizace větve z GitHubu

Pokud si chceme aktualizovat větev obsahem, který je na GitHubu (někdo jiný provedl nějaké úpravy, nebo si jen přenášíme mezi více PC rozdělanou práci), tak si nejdříve pomocí git branch zkontrolujeme, že jsme ve správné větvi, a poté jen stačí příkaz:
git pull origin nazev_vetve

Při pullování větve platí stejná pravidla jako při mergování a jeho konfliktech.

Stažení větve z GitHubu, která na lokálním PC ještě neexistuje

Pokud máte již u sebe nastavený repozitář, ale na GitHubu existuje větev, kterou u sebe lokálně nemáte vytvořenou, tak si ji potřebujeme vytvořit a stáhnout. Šlo by to samozřejmě provést pomocí příkazů, které jsme výše již použili, ale mohlo byt dojít k chybám a k tomuto účelu je vhodnější použít následující příkaz:
git fetch origin nazev_vetve_na_githubu:nazev_vetve_na_lokalu

Většinou doporučuji, aby oba názvy větví byly totožné. Pak už jen stačí se pomocí checkoutu přepnout na danou větev.

Vývojový cyklus

Cyklus práce pro vývojáře probíhá většinou stejným způsobem, jen občas je potřeba udělat nějaké úkony navíc. Předpokládejme, že máme již nastavený potřebný repozitář jak na GitHubu, tak na lokálním PC.

Nový úkol

  1. Byl mi přidělen nový úkol.
  2. Zkontroluji si, že jsem ve větvi “master”: git branch
  3. Pokud jsem v jiné větvi, přepnu se na “master”: git checkout master
  4. Aktualizuji “master” pro případ, kdyby tam někdo provedl změny:
     git pull origin master 
  5. Vytvořím novou větev: git checkout -b nova_vetev 
  6. Provedu změny v souborech.
  7. Oznámím Gitu, že jsem něco změnil: git add -A
  8. Změny zabalím do skupiny změn: git commit -m "zprava" 
  9. Odešlu změny na GitHub: git push origin nazev_vetve 

Pokračování na existujícím úkolu

  1. Byl mi přidělen již existující úkol.
  2. Zjistím, zda jsem na požadované větvi, či zda požadovaná větev u mě existuje:
     git branch 
  3. Pokud větev na lokálním PC neexistuje, stáhnu si ji z GitHubu:
     git fetch origin nazev_vetve:nazev_vetve 
  4. Pokud jsem na jiné větvi, přepnu se na požadovanou větev:
     git checkout nazev_vetve 
  5. Pokud jsme neprováděli bod 3, aktualizuji si větev z GitHubu:
    git pull origin nazev_vetve 
  6. Provedu změny v souborech.
  7. Oznámím Gitu, že jsem něco změnil: git add -A
  8. Změny zabalím do skupiny změn: git commit -m "zprava" 
  9. Odešlu změny na GitHub:  git push origin nazev_vetve
  10. Případně body 5 – 9 opakuji.

Závěr

Probrali jsme základní použití nástroje Git s využitím online serveru pro ukládání dat GitHub. Celý článek je jen krátkým výběrem toho, co lze v Gitu a v GitHubu provádět, a tak vás níže odkáži na odkazy, které s v případě potřeby můžete prostudovat.
